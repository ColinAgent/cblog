最近看了 Karpathy 的 LLM Wiki，我准备重新捡起自己的博客，重点记录这一年多深度研究 AI 工程时积累的所思所感。

这次重新写博客，不只是恢复一个发布渠道，而是想把博客和个人知识库真正融合起来：原始材料保存在 `raw` 目录里，发布时再让 AI 按不同平台的格式做转换、排版和必要的脱敏。这样既能保留最原始的思考痕迹，也能降低公开发布的摩擦。

## 最近一年的 AI 工程实践

大约从去年 4 月开始，我逐渐接触 AI 落地。当时比较火的是 Dify、n8n，网上到处都是 n8n 工作流、Dify AI 知识库之类的内容。我也是从 n8n 开始入门，同时接触了当时很热的 MCP。

我的第一次实践，是用 n8n 接入 MCP，实现用自然语言查询安全告警信息。整体流程大概是：用 MCP 接入 Wazuh indexer，自然语言入口放在飞书机器人里，用户和机器人对话后，消息转发到 n8n 进行 MCP 调用处理，最后再把 Wazuh 的查询结果通过飞书机器人返回给用户。

这个项目让我初步理解了 AI 在业务系统里的用法，也让我对它的大致工作原理有了直观认识。

第二次实践，是做一个企业级 AI 渗透测试系统，服务于某大型金融客户的定制场景。这个项目让我彻底看清了 Dify、n8n 这类工具在复杂企业级场景中的局限，也让我意识到当时很多主流 AI Agent 框架的问题。

比如 LangChain、LangGraph、Eino 这类框架，在很多场景里封装层级很深，概念也越来越复杂。我们从最开始使用这些框架，到后面逐渐抛弃这些框架，最后从 0 到 1 实现了一套完整的 AI Agent 通用框架，并把它落到了企业级业务场景里。

当时 AI 编程还没有现在这么强，很多代码基本都要手写，AI 生成的代码也远没有今天可靠。整个过程中走了非常多弯路，也重写了无数遍。

第三次实践，是用自己的 AI Agent 框架构建多 Agent 交易平台，并且在一次比赛或评测中获得了收益最高的结果。这个经历进一步验证了自研框架在复杂协同任务中的可行性。

第四次实践，则来自我自己的真实痛点：我不想在 Claude Code、Codex 这类编程 Agent 和 Markdown 软件之间不断切换。因此我开始做一款类似 Cursor、但更侧重 Markdown 和多工作空间的一体化客户端软件。目前这个项目的初版已经可以使用。

## Karpathy Wiki 给我的启发

最近看 Karpathy 的知识库结构时，`raw` 目录对我启发很大。

我准备采用类似的方式：把原始文章和想法先写到 `raw` 目录里；需要发布博客时，再让 AI 把 `raw` 里的原始文件转换到博客项目中，并根据博客的格式要求做相应的整理、排版和发布。

这样做有几个好处。

第一，源文件始终保持最原始的状态。原始材料不因为某个平台的格式要求而被污染，后续修改、版本管理和多平台复用都会更简单。

第二，发布过程可以自动化。AI 可以根据博客系统的 frontmatter、分类、标签、文件命名和构建要求自动处理，不需要每次手动调整格式。

第三，可以在发布前做一次安全检查。比如密钥、Token、内部地址、客户名称、私人对话、未公开项目等内容，都应该在发布版本里删除、匿名化或抽象化，而不是直接暴露出去。

## 我的新博客工作流

现在我的想法是：先在 `AGENTS.md` 里写清楚意图和流程。以后每次只要说一句“发布这篇文章到 cblog”，AI 就能自动执行完整流程：

1. 读取指定的 `raw` 原始文章；
2. 检查是否包含不适合公开的内容；
3. 转换成 cblog 需要的文章格式；
4. 必要时创建分类、复制图片并更新引用；
5. 运行构建验证；
6. 提交并推送到博客仓库，由 Cloudflare 自动部署。

后续如果要发布到微信公众号、X、小红书等平台，也可以继续扩展对应的 SOP。原始材料仍然只保留一份，平台版本由 AI 按需生成。

我会录制一个视频来演示这个流程的大概做法。对我来说，这不是单纯地“把文章发出去”，而是把写作、知识库、AI Agent 和发布系统连成一个新的个人工作流。
